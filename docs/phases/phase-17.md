# Phase 17: Executable Memory Manager

Phase 17 fills `nx86-jit` with the first executable-memory boundary for trusted
generated code. On Linux x86_64 it allocates anonymous memory writable, copies
generated bytes, switches the mapping to read+execute, and releases it on drop.

Apple Silicon and other unsupported hosts return a structured unsupported-host
error. This keeps local development deterministic while making the callable-code
exit criterion a Linux x86_64 responsibility.

The mmap, mprotect, munmap, and function-pointer conversion code is isolated in
the Linux platform module, with W^X permission changes and narrow call wrappers
for the generated function signatures used by the prototype. Executing a mapping
is an explicit unsafe API contract so call sites must prove the bytes are
trusted generated code with the requested ABI.

## Exit Criteria

- Empty code is rejected.
- Unsupported hosts report `UnsupportedHost` without attempting execution.
- Linux x86_64 can call trusted generated code through `ExecutableMemory`.
